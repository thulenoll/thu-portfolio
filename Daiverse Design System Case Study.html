01 — Context

DAI's teams were using thirteen applications. The brief: replace them with one.

DAI is a global development organization running social and economic projects across 150+ countries since 1970. Their teams were jumping between 10–13 different applications — SharePoint, Tammis, Oracle, and a long tail of internal tools — just to do the work. The brief from RCG Consulting (now Myridius): build a single unified platform — Daiverse — to replace the stack, and a design system that could hold it together over the next two years.

Over 90% of DAI's users work on desktop, so the first version focused there. Dark mode and mobile were planned for later, but I built the system to be ready for them from day one rather than having to bolt them on afterward.

The product never shipped. In early 2025, USAID was dismantled by the US government, and DAI's US operations — which relied on USAID funding — were forced to wind down before Daiverse reached production. What follows is the system itself: 18 months of work, built in enough detail to outlast the project that ended it.

Replacing thirteen apps with one means the system has to do more than look good. It has to be the shared foundation that keeps the new product consistent as it grows.
Notes from building the system
02 — Foundation

Turning a brand into a foundation a product could be built on.

DAI's brand guidelines — colors, type, charts, voice — were made for marketing, not software. I had to stretch them to handle a complex product without losing the look DAI had built over decades. The brand was a starting point, not a limit.

Colors, type, grid, spacing & elevation

I took DAI's core colors and built a full range out of each one, light to dark, as the base palette everything draws from. Proxima Nova carried over from the brand, set into a scale of text sizes (XXS to 6XL) with room for everything from dense data tables to big dashboard headers. Underneath it all, a 4-point grid lines up spacing, sizing, and padding, and a set of shadow styles gives cards, modals, and pop-ups a consistent sense of depth.

03 — Token architecture

Two layers, so the brand can change without breaking anything.

The system works in two layers. The bottom layer holds the raw values, the primitive tokens — every exact color and size. The top layer gives each value a job, the semantic tokens — background, text, accent, button state — and that's the only layer the components ever touch. Keeping them separate means the brand can change at the bottom without breaking anything built on top.

04 — Composition

A library built from what teams actually used.

I only added a component once a feature's design was final.

It's significantly easier to fill your design system with components you actually need and will use when you have real content and real use cases
Dan Mall · Design That Scales

DAI had no existing product to learn from — just an old marketing site. So I started from the brand's most-used patterns and grew the library as each feature shipped: buttons, navigation, widgets, charts. Light mode came first, with dark mode already wired up underneath so it would follow automatically instead of being added later.

05 — Distribution

A system people could actually use, not just look at.

Every value and component got a written description the moment it was created, so the rest of the team understood how to use it, not just what it looked like. ZeroHeight became the single home for all of it: the designs, the code, and the guidance on when to use what.

I worked directly with engineering to turn the components into code and bring that code back into the docs, so design and development stayed in step as the system grew. It was never just a Figma file — it was a shared agreement between designers and engineers.
