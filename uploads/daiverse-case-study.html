01 — Context

DAI's teams were using thirteen applications. The brief: replace them with one.

DAI is a global development organization running social and economic projects across 150+ countries since 1970. Their teams were jumping between 10–13 different applications — SharePoint, Tammis, Oracle, and a long tail of internal tools — just to do the work. The brief from RCG Consulting (now Myridius): build a single unified platform — Daiverse — to replace the stack, and a design system that could hold it together over the next two years.

Over 90% of DAI's users work on desktop, so MVP1 focused there. Dark mode and mobile were scoped to later phases, but the system had to be built to anticipate them from day one — through Figma variables, not through retrofit.

The product never shipped. In early 2025, USAID was dismantled by the US government, and DAI's US operations — which relied on USAID funding — were forced to wind down before Daiverse reached production. What follows is the system itself: 18 months of architecture, built to a level of detail that survives the project that ended it.

Replacing thirteen apps with one means the system has to do more than look good. It has to be the contract that holds the new product together as it grows.
Notes from the system architecture phase
02 — Foundation

Brand identity, extended into a system-ready foundation.

DAI's existing brand guidelines — colors, type, charts, voice — were built for marketing surfaces. The Daiverse system had to extend them for a complex product UI without losing the brand recognition the organization had built over decades. That meant treating the brand as a starting point, not a constraint.

Colors

I extended DAI's core hues into a primitive color set — each hue stepped from 50 to 900 in tints and shades. This becomes the raw token layer that every higher abstraction reads from.

FIG. 01DAI brand colors (left) extended into Daiverse primitives (right) — 50 → 900 scale, every shade the system needs.
Typography

Proxima Nova carried over from DAI's brand. I built a T-shirt-sized type scale (XXS → 6XL) to give the system enough range for everything from data tables to dashboard headers, while keeping the hierarchy enforceable.

FIG. 02Brand type (left) extended into a usable product type scale (right).
Grid · Spacing · Elevation

A 4-point grid system governs every spacing, sizing, and layout decision in the product — line heights, padding, margins, gutters. Elevation tokens define the depth language for cards, modals, and overlays. None of this is decorative — it's the math the system runs on.

FIG. 03Grid (left), elevation (right) — the spatial language of the system.
03 — Token architecture

Two collections, one contract.

Daiverse runs on a two-tier token system. Primitives hold raw values — every hex, every step on the scale. Semantic tokens map those primitives to intent — surface, foreground, accent, state. Components only ever read from the semantic layer. That separation is the contract: brand can evolve at the primitive layer without breaking anything downstream.

04 — Composition

A library built from real use cases, not anticipated ones.

Components were only added after the UI for each feature was finalized. The principle, borrowed from Dan Mall's Design That Scales: build the system from components teams actually use, not the ones they might. Avoid the graveyard of unused parts.

DAI had no prior digital product to inventory — just an outdated marketing site. So I worked from the brand's most-used patterns and grew the library outward as each feature shipped. Buttons, navigation, widgets, charts. Light mode first, dark mode wired through Figma variables so the second mode would inherit, not retrofit.

05 — Distribution

A system that ships, not one that decorates Figma.

Tokens and components got their descriptions in Figma the moment they were created — so designers using them inside the team had context, not just shapes. ZeroHeight became the published source of truth: the design specs, the engineering code snippets, and the usage guidance all in one place.

I worked directly with engineering to translate components into code and bring those snippets back into ZeroHeight, so design and development stayed in sync as the system grew. The system isn't a Figma file — it's the working contract between designers and engineers.

How it ended

Daiverse never shipped. The design system I built is intact; the product it was built for is not. What follows is a reading of what got made — and what remains usable, learnable, or reusable.
