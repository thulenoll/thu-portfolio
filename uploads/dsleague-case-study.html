01 — Context

Most fantasy apps are built around one sport. DSLeague is built around ten.

DSLeague — short for Decathlon Sports League — is a fantasy app where owners draft teams across ten sports, not players within one. Football in fall, basketball in winter, golf in spring, soccer year-round. The league runs continuously, so there's always a sport in season.

Ten sports means ten visual languages, ten audiences, ten color worlds. The brief: build one system that carries them all without flattening any of them.

The interesting design problem wasn't a screen. It was the system behind the screens — how to let ten sports stay distinct while one product feels coherent.
02 — The system

The system starts from sport.

Each of the ten sports earned a primitive color drawn from its existing identity — UEFA purple, MLB red, NHL black, NBA orange, and so on. Ten primitives became the foundation. Every other token in the system maps back to them.

Primitives feed semantic tokens: surface, foreground, accent, interactive states. Semantic tokens feed components. Components feed screens. The sport identity stays specific; the system stays consistent. Light mode and dark mode are baked in at the token layer — not retrofitted at the surface.

FIG. 02Semantic tokens — abstracting primitives into intent so every component reads the same vocabulary.

The component library is built from those tokens up: buttons, cards, navigation, data displays, charts, transaction lists. Documented as a usable library that the dev team can pull from directly — not a one-time artifact handed off and forgotten.

FIG. 03A portion of the DSLeague component library — token-driven, mode-aware, dev-ready.
03 — In product

A single product that feels like ten.

The result: each sport's surface carries its identity, but the navigation, layout, interactions, and structure are one product. A user moving from football to golf shouldn't feel like they switched apps — they should feel like they walked into a different room of the same building.

Outcome

DSLeague is a live product with real users. The design system isn't a portfolio exercise — it's the spine of a shipping platform, and every decision is one I have to live with as the team and the product grow.
